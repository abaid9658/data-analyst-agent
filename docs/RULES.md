# 📜 Rules

## AI Data Analyst Agent — Development Rules

These rules exist to prevent hallucinations, maintain quality, and ensure consistency across the entire codebase. Every contributor (human or AI) must follow these rules.

---

## 🚨 Critical Rules (Never Break)

1. **Never create unnecessary files** — every file must have a clear purpose tied to a feature.
2. **Never change existing APIs without updating API_SPEC.md** — and bumping the version if breaking.
3. **Never skip input validation** — all API inputs validated with Pydantic, all forms with Zod.
4. **Never use fake/mock data in production code** — use test factories in tests only.
5. **Never store plaintext secrets** — all credentials encrypted, all secrets in environment variables.
6. **Never use `any` type in TypeScript** — always define explicit types.
7. **Never execute user-provided code without sandboxing** — Python sandbox (RestrictedPython) required.
8. **Never run raw string SQL queries** — always use parameterized queries.
9. **Always ask before deleting files** — especially database migrations and existing APIs.
10. **Never skip error handling** — every async function must have try/catch.

---

## 📁 File Management Rules

- Every new file goes in the correct directory per FOLDER_STRUCTURE.md
- File names: `snake_case.py` (Python), `PascalCase.tsx` (React components), `kebab-case.ts` (utils)
- Never leave unused imports
- Never leave commented-out dead code (use git history instead)
- Keep files focused: one responsibility per file (SRP)

---

## 🔐 Security Rules

- All environment variables documented in `.env.example` (with fake values)
- Database credentials encrypted with Fernet before storage
- JWT tokens: short-lived access (15min) + refresh token rotation
- Rate limiting applied to all endpoints (stricter on auth)
- Prompt injection scanner runs before every LLM call
- SQL generated by AI always goes through the validator before execution
- File uploads: validate MIME type + magic bytes, not just extension
- No sensitive data in logs (mask emails, tokens, passwords)
- All sensitive operations recorded in audit_logs

---

## 🤖 AI Agent Rules

- Agent must always explain what tools it is using and why
- Agent must always show confidence level for predictions
- Agent must cite the data source for every insight
- Agent must not make up data — if data is missing, say so
- Agent must validate SQL before executing on user databases
- Agent must limit Python code to data analysis only (no system calls)
- Agent retry limit: max 3 retries on tool failure, then return error to user
- Agent must always provide key insights alongside charts and queries

---

## 🗃️ Database Rules

- Never modify migration files after they are merged to main
- Always create a new migration for schema changes
- Never drop columns — mark as deprecated, migrate data, drop in next release
- Always add indexes for foreign keys and common filter columns
- Never use raw SQL in application code — use SQLAlchemy ORM
- Always use async database sessions (asyncpg driver)

---

## 🧪 Testing Rules

- No feature merged without tests
- Minimum coverage thresholds enforced in CI (see CODING_STANDARDS.md)
- No mocking of the database in integration tests — use test database
- Factories for all test data (never hardcode UUIDs or emails in tests)
- Tests must be deterministic (no randomness without seeding)

---

## 📝 Documentation Rules

- Every public function has a docstring (Python) or JSDoc comment (TypeScript)
- All API changes reflected in API_SPEC.md before merging
- All user-facing changes get a CHANGELOG entry
- README updated when setup process changes
- Never delete documentation — deprecate and redirect instead

---

## 🔄 Git Rules

- Branch from `main`, never commit directly to `main`
- PR requires passing CI + 1 reviewer approval
- PR title must follow Conventional Commits format
- Squash commits before merging
- Tag releases with semantic version (`v1.2.3`)

---

## ⚡ Performance Rules

- All database queries must have explicit `LIMIT` on list queries
- No N+1 queries — use joins or eager loading
- Cache expensive operations (schema detection, ML outputs, chart specs)
- Heavy operations (ML training, report generation) go to Celery background tasks
- LLM responses must use streaming (SSE) — never wait for full completion
- File parsing always runs in background task — never in request handler

---

## 🚀 Deployment Rules

- All services must have health check endpoints
- Docker images must be production-hardened (non-root user, minimal base image)
- CI must pass before deploy to staging
- Staging must be validated before deploy to production
- Rollback plan documented before every major deployment
- Database migrations must be backward-compatible (no breaking changes in single deploy)
